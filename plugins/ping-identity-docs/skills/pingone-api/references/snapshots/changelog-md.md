# Changelog

Review changes and notices for the PingOne APIs or the associated SDKs, by year.

Subscribe to get automatic updates: [icon: rss-square, set=fa][PingOne Platform APIs Changelog RSS feed](changelog.xml)

## Important notices

### Support for opaque refresh tokens

On March 2, 2027, PingOne will issue only opaque refresh tokens, and JWTs will be deprecated for refresh tokens. You must update existing OpenID Connect (OIDC)-based applications to use opaque refresh tokens by March 1, 2027 to avoid your users being unable to access resources they need. Learn more in [Refresh tokens](foundations/authentication-concepts/access-tokens-and-id-tokens/token-claims.html#refresh-tokens).

### Key rotation policy for token signing

PingOne will only use signing keys from key rotation policies (KRPs) to sign ID tokens and access tokens, regardless of whether the audience for the access token is PingOne APIs or custom resources. Any OIDC-based applications not using the KRP will automatically update to use the default KRP by March 1, 2027. Learn more in [Key Rotation Policies](platform/certificate-management/key-rotation-policies.html).

### Mandatory `typ` header parameter for access tokens

On March 2, 2027, PingOne will always include the `typ` header property with the `at+jwt` value when minting access tokens, regardless of whether the intended audience is the UserInfo endpoint, PingOne APIs, or custom resources. To avoid service disruptions, enable the `typ` header property for each of your applications by this date. To resolve any issues, verify that each remains functional and work with the application team, the custom resource team, or both.

## 2026

### June

#### June 30

##### Bypass for users whose MFA setting is disabled

New PingOne MFA

By default, if a user's MFA setting has been set to disabled, they are not able to authenticate. If you want to allow users to bypass MFA if their MFA setting has been disabled, include the new property [blockUsersWithDisabledMfa](mfa/device-authentication-policy/create-device-authentication-policy.html) in your MFA policy with a value of `false`.

##### Block MFA authentication for disabled user accounts

New PingOne MFA

You can now use the new [blockDisabledUsers](mfa/device-authentication-policy/create-device-authentication-policy.html) parameter in device authentication policies to ensure that users whose accounts have been disabled cannot authenticate with MFA.

#### June 24

##### Data-based identity verification supports Australia Medicare Card

New PingOne Verify

You can now use the Australia Medicare Card to verify a user's identity with [Verify Data-Based Identity Verification](verify/verify-data-based-identity-verification.html#data-based-identity-verification-data-model).

#### June 15

##### API requests for working with PingID desktop app 1.0

New PingOne MFA PingID

With the release of PingID desktop app 1.0, there are now API requests for creating and activating a desktop device and for carrying out authentication with a desktop device. The series of steps involved in creating a device can be found in [PingID Desktop device](mfa/users/mfa-devices.html#pingid-desktop-device). The series of steps involved in authenticating with a desktop device can be found in [Device authentication with PingID Desktop](mfa/mfa-authentication/mfa-device-authentications.html#device-authentication-with-pingid-desktop).

#### June 12

##### PingAM Gateway support

New PingOne

The Gateway API service now supports PingAM connections. Learn more in [Create PingAM Connection](platform/gateway-management/gateways/create-pingam-gateway.html).

#### June 10

##### Group role assignments

New PingOne

You can now scope a role over a group, giving you more specific delegation for use cases where segmenting users by population isn't desirable. For details, refer to [Group Role Assignments](platform/group-role-assignments/group-role-assignments.html).

#### June 9

##### Custom notification sender - Twilio Messaging Service

New PingOne

You can now create a custom notification sender that uses a Twilio messaging service that you have defined in your Twilio account. You can find details in the [Create Phone Delivery Settings (Twilio Messaging Service)](platform/notifications/phone-delivery-settings/create-phone-delivery-settings-twilio-messaging-service.html) and [Update Phone Delivery Settings (select Twilio Messaging Service)](platform/notifications/phone-delivery-settings/update-phone-delivery-settings-twilio-messaging-service.html) examples.

##### Customer notification senders - JWT-based OAuth2 authentication

New PingOne

You can now create a custom PingOne notification sender for providers that require [JWT-based OAuth2 authentication](platform/notifications/phone-delivery-settings/create_phone_delivery_settings_oauth_jwt.html). This applies to both email and SMS/voice senders.

#### June 4

##### Excluding predictor rules

New PingOne Protect

For the three predictor types that use defined predictor rules (bot detection, suspicious device, traffic anomaly), you can now use the new [excludedDetectionRules array](protect/risk-predictors/update_bot_detection_predictor.html) to specify that one or more of the rules used by the predictor should be ignored when evaluating the level of risk.

You can also include specific predictor rules when [defining conditions in a composite predictor](protect/risk-predictors.html#predictor-rule-exclusion).

This capability is useful in situations where you find a specific rule is leading to legitimate authentication attempts being labeled as high risk.

### May

#### May 27

##### Password recovery notification template

New PingOne PingOne DaVinci

We've added a new notification template, `password_recovery_new`, which supports DaVinci flows using the PingOne Connector. Use this template to send a code to the user when they need to reset their password. Learn more in [Notifications Templates](platform/notifications/notifications-templates.html) and the [PingOne Connector](https://docs.pingidentity.com/connectors/p1_connector.html) documentation.

#### May 25

##### Setting device requirements for PingID mobile app

New PingOne PingID

A new endpoint, [deviceRequirements](platform/applications/device-requirements/update-device-requirements.html), has been introduced for specifying device requirement settings for using the PingID mobile app.

Previously, the device requirements could be specified along with more general PingID mobile app settings in a request to the `applications` endpoint.

If you have existing code that updated device requirements for the PingID mobile app, modify the code so that the device requirements are set using the new endpoint.

If you run such existing code without making these changes, you do not encounter an error but the device requirements are not updated.

#### May 19

##### Support for `typ` header parameter in JWT-based access tokens

New PingOne

PingOne now supports including the `typ` header parameter with the `at+jwt` value in JWT-based access tokens, whether the intended audience is the UserInfo endpoint, PingOne APIs, or custom resources. You can enable the `typ` header by setting the `includeTyp` property to `true`. Learn more in [Applications OIDC settings data model](platform/applications/applications-1.html#applications-oidc-settings-data-model).

On March 2, 2027, the `includeTyp` property will be removed, and PingOne will always include the `typ` header with the `at+jwt` value when minting access tokens for the UserInfo endpoint, PingOne APIs, or custom resources. To avoid service disruptions, validate and update your applications and custom resources with the `includeTyp` property before this date. Learn more in [Access token claims](foundations/authentication-concepts/access-tokens-and-id-tokens/token-claims.html#access-token-claims).

#### May 18

##### MFA device reports - users with MFA bypass

New PingOne MFA

When generating MFA device reports, you can now include the new value `bypassUntil` in the [fields array](mfa/users/mfa-reports.html#mfa-reports-data-model) to have the report include information about users who are currently allowed to bypass MFA.

##### Custom sender - new variables for country code and national number

New PingOne

When configuring a custom sender for SMS or voice, if your provider's API requires that the phone number be split into the country code and the rest of the number, you can use `${to.country-code}` and `${to.national-number}` instead of `${to}`, as described in `requests.body` and `requests.url` in [Custom provider phone delivery settings properties (excluding Twilio and Syniverse)](platform/notifications/phone-delivery-settings.html#custom-provider-phone-delivery-settings-properties-excluding-twilio-and-syniverse).

#### May 10

##### Custom senders - scopes for OAuth2

New PingOne

For custom notification senders where you have set the authentication method to OAUTH2, you can now use the new [authentication.scopes parameter](platform/notifications/phone-delivery-settings/create_phone_delivery_settings_oauth.html) to specify the capabilities that the issued token should have. This applies to both email and SMS/voice senders.

#### May 6

##### Password change notifications templates

New PingOne

We've added notification templates for password changes made on a user's PingOne account as a result of a DaVinci flow. `password_change_admin` is used when an administrator changes or resets a user's password, and `password_change_user` is used when the end user changes their own password. Learn more in [Notifications Templates](platform/notifications/notifications-templates.html) and in the [PingOne Connector](https://docs.pingidentity.com/connectors/p1_connector.html) documentation.

#### May 5

##### Moved `/v1` out of Postman variables and into the path of each endpoint

Improved PingOne

To make endpoint paths complete and to make the Postman path variables consistent, we moved the `/v1` URL segment out of the `{{apiPath}}` and `{{orchestratePath}}` Postman variables and into the endpoint in all content topics and in all public Postman collections that use those variables. All Postman `{{…​Path}}` variables are now only the scheme and domain of the endpoint. This has been a source of confusion for many readers.

In addition to the API documentation site, all collections available in Ping Identity's public Postman workspaces, [PingOne](https://www.postman.com/ping-identity/pingone/overview), [PingOne Use Case Library](https://www.postman.com/ping-identity/pingone-universal-services/overview), [PingOne Universal Services](https://www.postman.com/ping-identity/pingone-universal-services/overview), and [PingOne Early Access](https://www.postman.com/ping-identity/pingone-early-access/overview) have the new endpoints.

Remember to remove the `/v1` segment from the `{{apiPath}}` and `{{orchestratePath}}` variables in your Postman environments, and to update any scripts that reference these variables.

### April

#### April 30

##### The `redirect_uri` value is returned in `pi.flow` responses

Improved PingOne

This enhancement applies to applications configured with either PingOne authentication policies or DaVinci flow policies.

When an application includes `response_mode=pi.flow` and `redirect_uri=<URI>` in a request to the /as/authorize or /as/resume endpoints, PingOne now returns the `redirect_uri` value in the response. This occurs after the requested `redirect_uri` is validated against the Redirect URIs as configured in the application record in PingOne. This enables you to access the `redirect_uri` quickly without needing to check the requests.

To increase the visibility of critical information, we've changed the title of *Working With PingOne APIs* to [Before You Begin](before-you-begin/introduction.html) and moved its location in the table of contents to follow this Changelog document. In particular, you'll find here the discussion of [Forward Compatibility](before-you-begin/forward-compatibility.html).

#### April 9

##### Custom notification sender - Syniverse Channels

New PingOne

You can now create a custom notification sender that uses your defined Syniverse channels rather than individual phone numbers. You can find details in the [Create Phone Delivery Settings (Syniverse channels)](platform/notifications/phone-delivery-settings/create-phone-delivery-settings-syniverse-channels.html) and [Update Phone Delivery Settings (select Syniverse channels)](platform/notifications/phone-delivery-settings/update-phone-delivery-settings-syniverse-channels.html) examples.

#### April 7

##### Custom metadata properties for applications

New PingOne

We have added new endpoints to allow administrators to optionally manage custom metadata properties for applications. These properties are intended for administrative purposes, such as storing contact or other relevant information about the application. You can find details in [Application Metadata](platform/applications/application-metadata.html).

#### April 5

##### New status fields for push and OTP authentication methods

New PingOne MFA

The existing `pushEnabled` and `otpEnabled` fields indicate whether push and OTP were initially enabled for a device. Since the enabled/disabled status can change over time, two new objects, `pushStatus` and `otpStatus`, have been added for mobile devices to reflect the current device status for receiving pushes and the current device status for OTP authentication. You can find details in [Native Device Properties](mfa/users/mfa-devices.html#native-device-properties).

#### April 1

##### New notification templates for account creation and user verification

New PingOne

We've released a new notification template for account creation, which sends a notification to the user when an account is created using the PingOne Connector in DaVinci. We've also released a new notification template for user verification, in which you can customize the verification code notification templates to use in a DaVinci flow. Learn more in [Notifications Templates](platform/notifications/notifications-templates.html).

### March

#### March 30

##### On-premise credential signing keys for PingOne Credentials

New PingOne Credentials

PingOne Credentials enables you to sign credentials using your cryptographic keys that you hold on-premise, rather than using keys stored in PingOne. Refer to [Credential signing with on-premise keys](credentials/credential-signing-keys.html) for details.

#### March 29

##### Risk user profile data endpoint

New PingOne Protect

You can now erase all of the risk-related data that has been collected for a specific user, using the `riskUserProfile` endpoint. For details, refer to [Risk Data](protect/risk-data.html).

#### March 18

##### Deprecation of cardDesignTemplate in PingOne Credentials

Deprecated PingOne Credentials

PingOne Credentials no longer supports `cardDesignTemplate`, which is replaced by fixed templates, and is deprecated. The property `metadata.fields.isVisible`, which controls visibility of fields in `cardDesignTemplate`, is no longer required and is also deprecated. Existing credential types defined by a `cardDesignTemplate` will continue to work, but `cardDesignTemplate` cannot be updated. You can use [Update a Credential Type](credentials/credential-types/update-a-credential-type.html) to remove `cardDesignTemplate` and add a `visualizationTemplateData` object to such credential types.

#### March 12

##### Configurable attribute to locate users based on username tokens

New PingOne

Added the `userAttributeUsernameMatch` attribute for Microsoft 365 applications. This allows you to customize how the username in an STS Username token request is matched to a PingOne user profile, overriding the default email address match.

You can find details in the [Applications WS-Federation settings data model](platform/applications/applications-1.html#applications-ws-federation-settings-data-model).

#### March 10

##### Support for OIDC client-initiated backchannel authentication (CIBA)

New PingOne

PingOne now supports [OIDC client-initiated backchannel authentication](https://openid.net/specs/openid-client-initiated-backchannel-authentication-core-1_0.html) (CIBA). This enables an out-of-band authentication flow initiated by an end user from a consumption device, such as a point-of-sale system, and completed on the user's authentication device. The application must be configured with a `grantTypes` value of `ciba`. Learn more in [CIBA grant type](foundations/authentication-concepts/authorization-flow-by-grant-type/ciba-grant-type.html).

##### OAuth 2.0 token exchange

New PingOne

PingOne now supports OAuth 2.0 token exchange ([RFC 8693](https://www.rfc-editor.org/rfc/rfc8693.html)). This enables an application to present a subject token and optionally an actor token and receive an access token for a custom resource. The application must be configured with a `grantTypes` value of `token_exchange`. Learn more in [Token exchange grant type](foundations/authentication-concepts/authorization-flow-by-grant-type/token-exchange-grant-type.html).
